## Context

目前團隊訂餐仰賴人工在試算表中建立欄位、收集資料與手動統計。主要痛點是：
- 每次訂餐品項不同，菜單欄位需重建。
- 多人填寫時，容易漏填或重複提交。
- 飲料需要搭配甜度與冰塊，人工統計時易遺漏組合細節。

此變更目標是提供一個小型訂餐系統，允許主辦人先建立本次菜單，再分享表單給同事填寫，最後由系統自動輸出統計結果。

約束與前提：
- 優先支援單次活動（單一訂餐批次）的完整流程。
- 先以中小型團隊使用情境為主（例如 5-100 人）。
- 需保留未來擴充多次活動與歷史查詢的空間。

## Goals / Non-Goals

**Goals:**
- 提供菜單配置功能：餐點、飲料、甜度與冰塊選項可由主辦人自訂。
- 提供可分享的填單介面，讓多位同事可獨立提交訂單。
- 提供即時或可重算的統計結果：
  - 餐點數量統計。
  - 飲料依「品項 + 甜度 + 冰塊」組合統計。
- 建立最小可行資料模型，支援查詢與匯出統計資料。

**Non-Goals:**
- 不包含金流、付款分帳功能。
- 不包含供應商串接與自動下單。
- 不包含複雜權限系統（先以主辦人/填單者兩種角色概念處理）。
- 不優先處理跨活動彙總報表。

## Decisions

### Decision 1: 採用「活動（Order Session）」作為資料主軸
- 內容：每次訂餐建立一個活動，活動下包含菜單設定與多筆同事訂單。
- 理由：符合「每次菜單不同」需求，資料邊界清楚，便於封存與重算統計。
- 替代方案：以固定全域菜單為主軸。
- 為何不採用：固定菜單難以對應每次不同供應商與品項。

### Decision 2: 飲料客製化拆為標準欄位並以組合鍵統計
- 內容：飲料訂單儲存 drinkItem、sweetness、iceLevel 三個欄位；統計時以三者組合做 group-by。
- 理由：可精準回答「哪一種甜度冰塊組合」的需求，避免文字描述格式不一致。
- 替代方案：將飲料描述存成單一字串（例如「紅茶 半糖 少冰」）。
- 為何不採用：字串難以驗證與可靠統計，且容易出現同義詞問題。

### Decision 3: 同事識別採用「全局名單選取」策略
- 內容：主辦人預先維護全局同事名單；建立活動時綁定指定名單；同事填表時從下拉選單選取身份（非自由輸入）。
- 理由：完全避免重名與拼寫差異，同事身份確定無誤，主辦人掌握人員清冊。
- 替代方案：同事填表時自由輸入姓名或員編。
- 為何不採用：易出現拼寫誤差、重名問題、填表人員界定不清。

### Decision 4: 表單提交採「單人單筆」覆寫策略
- 內容：同一活動內，同一同事最多保留一筆最新提交記錄；如同事重複提交則覆寫舊提交。
- 理由：降低重複提交造成的超買風險，與團隊訂餐收單邏輯相符。
- 替代方案：允許同人多筆訂單並於後台手動去重。
- 為何不採用：增加主辦人負擔與統計不確定性。

### Decision 5: 統計採「可重算」而非僅儲存快照
- 內容：統計結果可由訂單資料隨時重算；必要時可額外快取。
- 理由：重算可確保資料修正後結果一致，避免快照不同步。
- 替代方案：每次提交即寫入累加統計。
- 為何不採用：回滾或修改訂單時易產生一致性問題。

### Decision 6: 活動可設定截止時間，逾時關閉填單
- 內容：每個活動可選填截止時間；若有設定，系統在截止後拒絕新提交或更新提交。
- 理由：符合團隊訂餐收單節奏，避免下單前仍有資料變動。
- 替代方案：永遠開放到主辦人手動關閉。
- 為何不採用：容易因人為疏忽造成晚交單混入。

### Decision 7: 提供兩種 CSV 匯出
- 內容：主辦人可匯出「最新有效訂單明細 CSV」與「統計結果 CSV（餐點統計與飲料組合統計）」。
- 理由：方便直接交給店家與留存核對，降低轉抄成本。
- 替代方案：僅提供畫面檢視，不提供匯出。
- 為何不採用：無法滿足常見的對外傳遞與留檔需求。

## Risks / Trade-offs

- [同事名稱不唯一導致覆寫錯誤] → Mitigation: 支援可選的唯一識別欄位（員編/Email）並加上基本格式驗證。
- [選項命名不一致造成統計分裂] → Mitigation: 甜度與冰塊改用主辦人預設選單，不允許自由輸入。
- [活動後期菜單異動影響已填資料] → Mitigation: 限制活動進行中僅可新增選項，刪除需二次確認並提示影響。
- [大量提交時統計延遲] → Mitigation: 先用即時計算，若資料量成長再加入快取或背景重算機制。

## Migration Plan

- Phase 1: 建立活動與菜單資料模型（餐點/飲料/甜度/冰塊）。
- Phase 2: 建立填單 API 與前端表單，完成提交與覆寫規則。
- Phase 3: 建立統計 API 與檢視頁，輸出餐點與飲料組合統計。
- Phase 4: 小規模內部試用，蒐集邊界案例並修正。

Rollback strategy:
- 若新流程異常，主辦人可回退到匯出原始訂單明細再人工整理。
- 系統層面可關閉統計頁，保留收單功能，避免全面不可用。
